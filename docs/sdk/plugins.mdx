The SDK uses the `AgentExtension` interface for plugins. `AgentPlugin` is a public alias exported by `@clinebot/core`.

Plugin extension points:

| Extension point | Use when you want to... |
|-----------------|--------------------------|
| Tool | Let the model call an action |
| Plugin | Package tools, commands, message builders, providers, automation events, and hooks |
| Hook | Run lifecycle logic or policy checks |

## Tools vs Plugins vs Hooks

| Need | Use |
|------|-----|
| Query a database, call an API, run a domain action | Tool |
| Bundle several tools into a reusable module | Plugin |
| Log runs, collect metrics, enforce policy | Hook handler |
| Block dangerous tool calls | Hook or approval policy |
| Register slash commands or providers | Plugin |

For tool definitions, see [Tools](/sdk/tools).

## Extension Example

```typescript
import { createTool, type AgentExtension } from "@clinebot/core"

const databasePlugin: AgentExtension = {
  name: "database-tools",
  manifest: { capabilities: ["tools", "hooks"], hookStages: ["tool_call_before"] },

  setup(api, ctx) {
    api.registerTool(
      createTool({
        name: "query_database",
        description: "Run a read-only SQL query against the application database.",
        inputSchema: {
          type: "object",
          properties: {
            sql: { type: "string", description: "SQL SELECT query to execute" },
          },
          required: ["sql"],
        },
        execute: async (input) => db.query(input.sql),
      })
    )
  },

  onToolCall(context) {
    console.log("tool call", context)
  },
}
```

Register with `Agent` runtime plugins:

```typescript
const agent = new Agent({
  // ...config
  plugins: [databasePlugin],
})
```

Use `plugins` with `Agent` / `AgentRuntime`. Use `extensions` with `ClineCore` session config.

Register with `ClineCore`:

```typescript
await cline.start({
  prompt: "Analyze customer records",
  config: {
    // ...model/runtime config
    extensions: [databasePlugin],
  },
})
```

## File-Based Plugins

`ClineCore` supports plugin module paths via `pluginPaths` in session config:

```typescript
await cline.start({
  prompt: "Analyze this project",
  config: {
    // ...model/runtime config
    pluginPaths: ["/absolute/path/to/plugin.ts"],
  },
})
```

Plugin files export an `AgentExtension`.

## Installing Plugins via CLI

The `clite plugin install` command installs plugins from npm packages, git repositories, or local paths into the Cline plugin directory. Installed plugins are automatically discovered by the runtime.

```bash
clite plugin install ./my-plugin                        # local directory
clite plugin install https://github.com/owner/repo.git  # git repository
clite plugin install npm:@scope/my-plugin               # npm package
```

Use `--force` to replace an existing install and `--cwd <path>` to install into a project-specific plugin directory instead of the global one.

To be installable, a plugin's `package.json` should declare entry points in a `cline` field:

```json
{
  "cline": {
    "plugins": [
      {
        "paths": ["./index.ts"],
        "capabilities": ["tools", "hooks"]
      }
    ]
  }
}
```

If `cline.plugins` is absent, the installer auto-discovers `.ts` and `.js` entry files.

See [Plugins (Customization)](/customization/plugins) for the full manifest format and directory layout.

## Hook Stages

Hook stages include:

```txt
input
runtime_event
session_start
run_start
iteration_start
turn_start
before_agent_start
tool_call_before
tool_call_after
turn_end
stop_error
iteration_end
run_end
session_shutdown
error
```

Common stages:

| Stage | Use for |
|-------|---------|
| `before_agent_start` | Inject context or modify prompt/messages |
| `run_start` | Logging, timers, rate limits |
| `tool_call_before` | Audit or block tool calls |
| `tool_call_after` | Log results, trigger side effects |
| `run_end` | Metrics, notifications, cleanup |
| `error` | Error reporting |

## Hook Policies

Hook policies control execution behavior:

| Field | Meaning |
|-------|---------|
| `mode` | `"blocking"` or `"async"` |
| `timeoutMs` | Hook timeout |
| `retries` | Retry count |
| `retryDelayMs` | Delay between retries |
| `failureMode` | `"fail_open"` or `"fail_closed"` |
| `maxConcurrency` | Concurrent hook executions |
| `queueLimit` | Queue size before dropping |

Use `fail_closed` for policy-enforcement hooks where bypassing the hook is unsafe.
