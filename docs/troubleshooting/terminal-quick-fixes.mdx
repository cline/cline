## Try This First: Use the Built-In Background Terminal Runtime

Cline now uses the same background terminal runtime everywhere, including VS Code. If terminal behavior seems stuck or inconsistent, start with these fixes:

1. Click **Settings** (top right of Cline chat)
2. Go to **Terminal Settings**
3. Turn off **Enable aggressive terminal reuse** if commands seem to inherit the wrong working directory or stale terminal state

This keeps each command on the unified background runtime while avoiding the reuse behavior that causes most residual terminal issues.

---

## Other Fixes

-   **Disable terminal reuse**
    1. Within Cline, left-click the **Settings** button in the top right-hand corner of the chat window
    2. Once in the **Settings** window, left-click the **"Terminal Settings"** tab from the left-hand column
    3. Look for **"Enable aggressive terminal reuse"**, and **uncheck** this option

-   **Restart the task after changing your shell environment**

    If you recently changed your shell startup files (`.zshrc`, `.bashrc`, `.profile`, etc.), start a fresh Cline task so new background commands pick up the updated environment.

-   **Keep shell startup fast and non-interactive**

    Background commands work best when your login shell can start without prompts, blocking TUI tools, or slow network-dependent startup scripts.

## Platform-Specific Fixes

### macOS + Oh-My-Zsh

```bash
# Create minimal config for VSCode
echo 'export TERM=xterm-256color' > ~/.zshrc-vscode
echo 'export PAGER=cat' >> ~/.zshrc-vscode
```

### Windows PowerShell

```powershell
# Run as Administrator
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

### WSL

-   Open folder from WSL: `code .`
-   Relaunch Cline after opening the WSL workspace so new commands run inside the Linux environment
-   Keep your WSL shell startup files lightweight and non-interactive
