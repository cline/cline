Hooks let you run code at specific points in the agent's lifecycle. Unlike [extensions](/cline-sdk/extensions) (which add capabilities like tools), hooks are designed for observation and policy enforcement: logging, auditing, blocking dangerous operations, modifying the system prompt, or triggering external workflows.

## Hook Lifecycle

Hooks fire at these stages of the agent's execution:

```
session_start
  run_start
    before_agent_start          <- modify system prompt, messages
      iteration_start
        turn_start
          tool_call_before      <- inspect/block tool calls
          tool_call_after       <- inspect tool results
        turn_end
      iteration_end
    run_end
  session_shutdown
```

## Available Hooks

| Hook | When it fires | Common uses |
|------|--------------|-------------|
| `session_start` | Session is created | Initialize external resources |
| `before_agent_start` | Before the first LLM call | Modify system prompt, inject context |
| `run_start` | `run()` or `continue()` begins | Log user message, start timers |
| `iteration_start` | Each loop iteration begins | Track iteration count |
| `turn_start` | LLM API call begins | |
| `tool_call_before` | Before a tool executes | Block dangerous operations, audit |
| `tool_call_after` | After a tool completes | Log results, trigger side effects |
| `turn_end` | LLM response fully processed | |
| `iteration_end` | Loop iteration completes | |
| `run_end` | `run()` or `continue()` finishes | Log result, stop timers |
| `session_shutdown` | Session is being cleaned up | Clean up external resources |
| `error` | An error occurs | Error reporting, alerting |
| `stop` | Agent is stopping | |

## Using Hooks in Extensions

The simplest way to use hooks is through the [extension](/cline-sdk/extensions) lifecycle methods:

```typescript
const auditExtension: AgentExtension = {
  name: "audit-log",
  manifest: { capabilities: ["hooks"] },

  onRunStart({ userMessage }) {
    auditLog.write({ event: "run_start", message: userMessage, timestamp: Date.now() })
  },

  onToolCall({ call }) {
    auditLog.write({ event: "tool_call", tool: call.name, input: call.input })
  },

  onRunEnd({ result }) {
    auditLog.write({
      event: "run_end",
      finishReason: result.finishReason,
      iterations: result.iterations,
    })
  },
}
```

## File-Based Hooks

ClineCore discovers hook scripts from `.cline/hooks/` in your project root and `~/.cline/data/settings/hooks/` for global hooks. These are shell scripts or executables that run in a separate process.

```
.cline/hooks/
  before_agent_start.sh
  tool_call_before.sh
  run_end.sh
```

Each hook script receives event data via stdin as JSON and can return a response via stdout:

```bash
#!/bin/bash
# .cline/hooks/tool_call_before.sh
# Block any rm -rf commands

INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.toolName')
COMMAND=$(echo "$INPUT" | jq -r '.input.command // empty')

if [[ "$TOOL_NAME" == "bash" && "$COMMAND" == *"rm -rf"* ]]; then
  echo '{"action": "block", "reason": "Destructive commands are not allowed"}'
  exit 0
fi

echo '{"action": "allow"}'
```

## Hook Policies

Hooks can be configured with failure policies that determine what happens if a hook throws an error or fails:

| Policy | Behavior |
|--------|----------|
| `fail_open` | Hook failure is logged but execution continues |
| `fail_closed` | Hook failure stops the agent |

The default policy depends on the hook type. Observational hooks (like logging) default to `fail_open`. Policy-enforcement hooks (like `tool_call_before`) should use `fail_closed` if blocking is critical.

## Modifying the System Prompt

The `before_agent_start` hook is a seam for modifying the system prompt or initial messages before the agent makes its first LLM call:

```typescript
const contextInjectionExtension: AgentExtension = {
  name: "context-injection",
  manifest: { capabilities: ["hooks"] },

  // This fires before the first LLM call
  onBeforeAgentStart({ systemPrompt, messages }) {
    // Inject additional context into the system prompt
    const additionalContext = `
Current project: ${process.env.PROJECT_NAME}
Current sprint: ${process.env.SPRINT_ID}
Team guidelines: Always write tests for new functions.
`
    return {
      systemPrompt: systemPrompt + "\n\n" + additionalContext,
      messages,
    }
  },
}
```

## Practical Examples

### Slack Notification on Completion

```bash
#!/bin/bash
# .cline/hooks/run_end.sh
INPUT=$(cat)
FINISH_REASON=$(echo "$INPUT" | jq -r '.finishReason')
ITERATIONS=$(echo "$INPUT" | jq -r '.iterations')

curl -X POST "$SLACK_WEBHOOK_URL" \
  -H "Content-Type: application/json" \
  -d "{\"text\": \"Agent finished: ${FINISH_REASON} (${ITERATIONS} iterations)\"}"
```

### Cost Tracking

```typescript
const costTracker: AgentExtension = {
  name: "cost-tracker",
  manifest: { capabilities: ["hooks"] },

  onRunEnd({ result }) {
    const cost = calculateCost(result.usage, result.model)
    database.insert("agent_costs", {
      sessionId: result.sessionId,
      cost,
      tokens: result.usage.inputTokens + result.usage.outputTokens,
      timestamp: new Date(),
    })
  },
}
```

### Rate Limiting

```typescript
const rateLimiter: AgentExtension = {
  name: "rate-limiter",
  manifest: { capabilities: ["hooks"] },

  onRunStart() {
    const recentRuns = getRunsInLastMinute()
    if (recentRuns >= 10) {
      throw new Error("Rate limit exceeded: maximum 10 runs per minute")
    }
  },
}
```
